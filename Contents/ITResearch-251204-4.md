# JAVA와 Kotlin

<br>

### 자바(JAVA) vs 코틀린(Kotlin) 비교
| 분류 | Java | Kotlin |
| --- | --- | --- |
| 등장 배경 | 1995년 출시, 오랜 역사, 엔터프라이즈 환경의 표준 | 2011년 JetBrains 출시, Java의 단점 개선을 목표 |
| 실행환경 | JVM (Java Virtual Machine) | JVM (Java와 100% 상호 운용 가능) |
| 코드 간결성 | 상대적으로 장황함 | 매우 간결함 (Getter/Setter, 생성자 등 자동 생성) |
| Null처리 (안정성) | NullPointerException (NPE) 발생 위험 높음 | Null Safety 기능 기본 제공 (컴파일 시점에 오류 방지) |
| 비동기 처리 | 스레드 기반, 복잡함 | 코루틴(Coroutines) 기본 제공, 경량 스레드로 간결하고 효율적 |
| 트렌드/채택 | 여전히 거대한 레거시 및 엔터프라이즈의 표준 | 스타트업 및 현대적 백엔드 시스템에서 높은 채택률 |

<br>

### 코틀린의 등장 배경
- JAVA의 지배력 : JAVA는 오랫동안 백엔드의 표준이었지만, 시간이 지나면서 문법의 장황함과 고질적인 NPE(NullPointerException) 문제가 개발 생산성을 저해하기 시작했음
- Kotlin의 목표 : Java의 단점을 보완하고, 더 안전하며, 더 간결하고 생산적인 JVM 언어를 만드는 것


<br>

### 핵심장점 1 : 뛰어난 간결성과 생산성
- Boilerplate Code 최소화 : Java에서 흔히 작성하는 Data Class, 생성자, Getter, Setter 등의 반복적인 코드가 코틀린에서는 data class 키워드 등으로 자동 생성 된다.
- 함수형 프로그래밍 지원 : 람다식, 고차 함수 등을 자연스럽게 지원하여 데이터 가공 및 처리가 매우 간결하고 효율적임

<br>

### 핵심장점 2 : Null Safety를 통한 안정성
- Null의 위험 : Java는 Null 값을 허용하여, 런타임에 NPE가 발생해 서비스에 장애를 일으킬 가능성이 높다
- Kotlin의 해결 :
  - 변수를 기본적으로 Non-null로 간주한다.
  - Null이 가능함을 명시적으로 '?' 연산자를 붙여 선언해야 한다. (ex: var name: String?)
  - ?. (Safe Call), ?: (Elvis Operator) 등을 통해 컴파일 시점에 Null 참조를 검사하고 안전하게 처리하도록 강제한다. -> 버그 감소에 큰 기여
 
<br>

### 백엔드에서의 경쟁력 : 코루틴(Coroutines)
- 비동기 처리 : 백엔드는 I/O 작업(DB 접근, 외부 API 호출)이 많아 비동기 처리가 매우 중요하다.
- 코루틴(Coroutines): 코틀린은 경량 스레드인 코루틴을 기본으로 제공하여, Java의 복잡한 비동기 방식(스레드 관리)보다 훨씬 효율적이고 직관적으로 비동기 코드를 작성할 수 있게 한다. -> 높은 동시성 처리 능력

<br>

### 코틀린 우대/필수 채용 기업
1. 핀테크 / 블록체인 기업
  - 특징 : 금융 서비스의 특성상 높은 안정성(Null Safety)과 신속한 동시성 처리(코루틴)가 필수적임.
  - 대표사례 : 코인원 등의 금융 및 블록체인 관련 기업에서 Kotlin 기반의 백엔드 개발자를 채용하는 공고가 확인됨.

2. 이커머스 / 플랫폼 서비스 기업
   - 크몽, 무신사, 클래스101 등 ...
4. IT 기술 기반 스타트업 솔루션 기업
   - 오누이, 트래블월렛, 엔드앤드코리아, 비욘드메디슨 등
